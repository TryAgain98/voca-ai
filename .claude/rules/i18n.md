# Internationalization (i18n)

## Stack

- `next-intl` v4. Locales: `en` (default), `vi`.
- Translation files: `messages/en.json` and `messages/vi.json`.
- URL structure: `/{locale}/route` — locale is always the first segment.

---

## Zero Hardcoded Strings in JSX

**Every user-visible string must go through `useTranslations` or `getTranslations`.** No exceptions.

```tsx
// ✅
const t = useTranslations('Lessons')
return <h1>{t('title')}</h1>

// ❌
return <h1>Lessons</h1>
return <h1>Bài học</h1>
```

Strings that are NOT translated (and never should be): log messages, error thrown to the console, code comments, variable names, URL paths.

---

## Always Update Both Files Simultaneously

When adding any translation key, add it to **both** `en.json` and `vi.json` in the same edit. Never leave one file ahead of the other.

```json
// messages/en.json
{
  "Lessons": {
    "title": "Lessons",
    "empty": "No lessons yet"
  }
}

// messages/vi.json — must be updated in the same change
{
  "Lessons": {
    "title": "Bài học",
    "empty": "Chưa có bài học nào"
  }
}
```

If a translation is temporarily unknown, use the English string as a placeholder — but add a `// TODO: translate` comment in the JSON and track it. Never leave a key missing from one file.

---

## Namespace Convention

Use PascalCase namespaces that match the feature or page:

| Feature                     | Namespace |
| --------------------------- | --------- |
| Lessons page                | `Lessons` |
| Auth pages                  | `Auth`    |
| Shared UI (buttons, labels) | `Common`  |
| Navigation                  | `Nav`     |
| Error messages              | `Errors`  |

Avoid one giant flat namespace — it becomes unmaintainable.

## Server vs Client Usage

**Server components / layouts:**

```ts
import { getTranslations } from 'next-intl/server'

export default async function LessonsPage() {
  const t = await getTranslations('Lessons')
  return <h1>{t('title')}</h1>
}
```

**Client components:**

```ts
'use client'
import { useTranslations } from 'next-intl'

export function LessonCard() {
  const t = useTranslations('Lessons')
  return <p>{t('empty')}</p>
}
```

Never import `useTranslations` in a Server Component or `getTranslations` in a Client Component.

---

## Locale Switching

Switch locale by replacing the locale segment — never hardcode a locale:

```ts
const pathname = usePathname()
const newPath = pathname.replace(`/${locale}`, `/${next}`)
router.push(newPath)
```

Never construct paths like `/en/lessons` or `/vi/lessons` with string literals.

---

## Dynamic Values in Translations

Use ICU message format for interpolation, plurals, and conditionals — never concatenate strings:

```json
{
  "lessonCount": "{count, plural, =0 {No lessons} =1 {1 lesson} other {# lessons}}"
}
```

```tsx
// ✅
t('lessonCount', { count: lessons.length })
// ❌
`${lessons.length} lessons`
t('lesson') + 's'
```

---

## Type Safety

`next-intl` generates types from your message files. If a key doesn't exist in the JSON, TypeScript will error. This means:

- Never use `t('some.key' as string)` to bypass the type check.
- If a key is dynamic, define all variants in the JSON and use a union type for the key parameter.
